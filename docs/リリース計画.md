# コミュニティプラットフォーム リリース計画

## 前提

- **全28モジュール・116テーブル・53画面**を一括リリース
- **決済機能（Stripe/Komoju）は対象外**（注文記録のみ）
- **開発体制**: 1名 + Claude Code
- **目標**: なるべく早くリリース

---

## 外部サービス登録・セットアップ

各フェーズで使用する外部サービスを、必要になるタイミング順に整理する。
**各サービスは、該当フェーズの開始前までにセットアップを完了しておくこと。**

| #   | サービス          | 用途                       | 必要フェーズ | 無料枠                                                |
| --- | ----------------- | -------------------------- | :----------: | ----------------------------------------------------- |
| 1   | GitHub            | リポジトリ・CI/CD          |   Phase 0    | 無制限（パブリック）/ 2,000分 Actions（プライベート） |
| 2   | Supabase          | PostgreSQL DB              |   Phase 0    | DB 500MB・無制限APIリクエスト                         |
| 3   | Railway           | NestJS・Redis ホスティング |   Phase 0    | $5/月クレジット・512MB RAM                            |
| 4   | Vercel            | Next.js ホスティング       |   Phase 0    | 100GB帯域・サーバーレス関数制限あり                   |
| 5   | Sentry            | エラー監視                 |   Phase 0    | 月5,000イベント                                       |
| 6   | Cloudflare R2     | ファイルストレージ         |   Phase 1    | 10GB保存・月100万リクエスト（クラスB）                |
| 7   | Resend            | メール配信                 |   Phase 1    | 月3,000通・1ドメイン                                  |
| 8   | Cloudflare Stream | 動画配信                   |   Phase 7    | なし（$5/1,000分保存 + $1/1,000分視聴）               |
| 9   | LINE Developers   | LINE連携                   |   Phase 10   | 月200通（無料メッセージ）                             |
| 10  | Slack             | エラー通知連携             |   Phase 12   | 無料（Webhook）                                       |

### 1. GitHub（Phase 0 開始前）

- リポジトリ作成（プライベート推奨）
- `main` / `dev` ブランチ作成
- Branch protection rules 設定（PRレビュー必須等）

### 2. Supabase（Phase 0）

- アカウント作成 → 新規プロジェクト作成（リージョン: 東京 `ap-northeast-1`）
- ダッシュボードから pgroonga 拡張を有効化（Database → Extensions）
- 接続情報を取得:
  - `DATABASE_URL`（Prisma 用 Connection pooling URL）
  - `DIRECT_URL`（マイグレーション用 Direct connection URL）
- パスワード（プロジェクト作成時に設定）を安全に保管

### 3. Railway（Phase 0）

- アカウント作成 → 新規プロジェクト作成
- **NestJS サービス**: GitHub リポジトリを接続、`apps/api` をルートディレクトリに設定
- **Redis サービス**: テンプレートから Redis を追加
- 取得する情報:
  - `REDIS_URL`（Redis の内部接続URL）
  - Railway のデプロイ先URL（API_URL として使用）

### 4. Vercel（Phase 0）

- アカウント作成 → GitHub リポジトリをインポート
- フレームワークプリセット: Next.js、ルートディレクトリ: `apps/web`
- 取得する情報:
  - デプロイ先URL（`NEXT_PUBLIC_APP_URL`）
- 環境変数に `NEXT_PUBLIC_API_URL`（Railway のAPI URL）を設定

### 5. Sentry（Phase 0）

- アカウント作成 → Organization 作成
- **2つのプロジェクト**を作成:
  - `community-web`（Next.js / React）
  - `community-api`（Node.js / NestJS）
- 取得する情報:
  - `SENTRY_DSN`（web 用・api 用それぞれ）
  - `SENTRY_AUTH_TOKEN`（ソースマップアップロード用）
  - `SENTRY_ORG` / `SENTRY_PROJECT`

### 6. Cloudflare R2（Phase 1 開始前）

- Cloudflare アカウント作成（未登録の場合）
- R2 を有効化 → バケット作成（例: `community-files`）
- CORS ポリシーを設定（フロントエンドからの直接アップロード許可）
- APIトークン作成（R2 の読み書き権限）
- 取得する情報:
  - `CLOUDFLARE_ACCOUNT_ID`
  - `R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY`
  - `R2_BUCKET_NAME`
  - `R2_PUBLIC_URL`（カスタムドメインまたは `r2.dev` サブドメイン）

### 7. Resend（Phase 0 中に着手 → Phase 1 で使用）

- アカウント作成
- 送信ドメインを追加 → DNS レコード（SPF / DKIM / DMARC）を設定・検証
  - **ドメイン検証に最大48時間かかるため、Phase 0 中に着手すること**
- APIキー作成
- 取得する情報:
  - `RESEND_API_KEY`
  - `MAIL_FROM_ADDRESS`（例: `noreply@yourdomain.com`）

### 8. Cloudflare Stream（Phase 7 開始前）

- Cloudflare ダッシュボードで Stream を有効化（有料: $5/1,000分保存）
- Webhook URL を設定（動画のエンコード完了通知用）
- 署名付きURL 用の署名キーを作成
- 取得する情報:
  - `CLOUDFLARE_STREAM_API_TOKEN`
  - `CLOUDFLARE_STREAM_SIGNING_KEY_ID` / `CLOUDFLARE_STREAM_SIGNING_KEY_JWK`
  - `CLOUDFLARE_STREAM_WEBHOOK_SECRET`

### 9. LINE Developers（Phase 10 開始前）

- LINE Developers Console でプロバイダー作成
- Messaging API チャネルを作成
- Webhook URL を設定（NestJS の LINE webhook エンドポイント）
- 取得する情報:
  - `LINE_CHANNEL_ID` / `LINE_CHANNEL_SECRET`
  - `LINE_CHANNEL_ACCESS_TOKEN`

### 10. Slack Incoming Webhook（Phase 12）

- Slack ワークスペースで Incoming Webhook を作成
- Sentry → Slack 連携を設定
- 取得する情報:
  - `SLACK_WEBHOOK_URL`

### セットアップ時の注意事項

1. **Resend のドメイン検証は早めに**: DNS 反映に最大48時間かかるため、Phase 0 中にドメイン追加・DNS設定を行う
2. **Cloudflare Stream は有料**: 無料枠なし。Phase 7 開始時に有効化すればよい（それまで費用は発生しない）
3. **Railway の無料枠に注意**: $5/月のクレジットのみ。開発中はローカル Docker を主に使い、Railway は staging/CI 用にする
4. **本番用は別プロジェクト**: Phase 12 で本番環境用の Supabase / Railway / Vercel プロジェクトを別途作成する（開発用と本番用を分離）

## Phase 0: プロジェクト基盤構築（3-4日）

### 目的

モノレポ・開発環境・CI/CD を整備し、全フェーズの土台を作る。

### タスク

**0.1 モノレポ構成（Turborepo + pnpm workspaces）**

- `apps/web` — Next.js フロントエンド
- `apps/api` — NestJS バックエンド
- `packages/shared` — 共有型定義・Zodスキーマ・定数・ユーティリティ
- ルート `tsconfig.base.json` でTypeScriptプロジェクト参照を設定

**0.2 Next.js 初期化（`apps/web`）**

- App Router / TypeScript / Tailwind CSS で作成
- shadcn/ui 導入、TanStack Query プロバイダ設定

**0.3 NestJS 初期化（`apps/api`）**

- `@nestjs/config`, `@nestjs/jwt`, `@nestjs/passport`, `@nestjs/swagger`, `@nestjs/bullmq`, `@nestjs/websockets` 導入
- Prisma 設定、環境変数バリデーション（Zod）

**0.4 共有パッケージ（`packages/shared`）**

- `tsup` でビルド設定
- 基本型定義: `ApiResponse<T>`, `PaginatedResponse<T>`, 共通enum

**0.6 開発環境**

- Docker Compose: PostgreSQL 16 (pgroonga) + Redis 7
- `.env.example` 作成
- ESLint + Prettier + Husky + lint-staged

**0.7 CI/CD**

- GitHub Actions: PR時 lint/type-check/test、dev マージ時 staging デプロイ、main マージ時 production デプロイ
- Vercel（web）+ Railway（api）の接続

### 成果物

`pnpm dev` でフロント(3000) + バックエンド(3001) が起動する状態

## Phase 1: 認証・コアシェル（7-10日）

### 目的

認証システム、ユーザー管理、コミュニティ設定、アプリケーションシェル（レイアウト・ナビゲーション）を構築。ログインして画面遷移できる状態を作る。

`users`, `social_accounts`, `login_histories`, `refresh_tokens`, `password_reset_tokens`, `feature_settings`, `member_ranks`, `permission_settings`, `app_settings`, `categories`, `tags`, `files`, `user_profiles`, `user_interests`, `user_languages`, `user_affiliations`, `user_public_info`

### バックエンド

- **認証モジュール**: 登録・ログイン・リフレッシュ・ログアウト・パスワードリセット（Resendでメール送信）
- **JWT戦略**: Passport.js、`JwtAuthGuard`、`RolesGuard`
- **ユーザーモジュール**: プロフィールCRUD、ユーザー一覧（ページネーション・検索）
- **コミュニティ管理**: 機能トグルCRUD、権限設定CRUD、アプリ設定CRUD
- **機能チェックミドルウェア**: 無効モジュールへのリクエストを403で拒否
- **ファイルアップロード**: Cloudflare R2 へのアップロード、Sharp でサムネイル生成
- **シードデータ**: `feature_settings`（28機能フラグ）、デフォルト `member_ranks`

### フロントエンド

- **アプリシェル**: ヘッダー（ロゴ・通知ベル・検索・ユーザーメニュー）+ サイドバー（機能トグルに連動）+ メインコンテンツ
- **認証画面（3画面）**: ログイン、新規登録（3ステップ）、パスワードリセット
- **設定画面（3画面）**: コミュニティ設定、権限設定、デザイン設定
- **メンバー画面（2画面）**: メンバー一覧、メンバー詳細
- **共通コンポーネント**: DataTable、FileUpload、RichTextEditor（Tiptap）、ConfirmDialog

### 成果物

ユーザー登録・ログイン可能。ナビゲーションシェルが動作し、管理者が機能トグル・権限を設定できる。

## 横断的な方針

### DBマイグレーション戦略

- Phase ごとにPrismaマイグレーションを1つ生成
- pgroongaインデックスはカスタムSQLで別途管理
- 本番適用前に必ずステージングで検証

### テスト戦略

| レイヤー    | ツール                   | 対象                                     | タイミング   |
| ----------- | ------------------------ | ---------------------------------------- | ------------ |
| Unit        | Jest                     | サービス・ユーティリティ・バリデーション | 各PR         |
| Integration | Jest + テストDB          | APIエンドポイント・DB操作                | 各PR         |
| Component   | Vitest + Testing Library | Reactコンポーネント                      | 各PR         |
| E2E         | Playwright               | 主要ユーザーフロー                       | Phase 11以降 |

### アーキテクチャパターン（全フェーズ共通）

1. **NestJSモジュール**: `module.ts` / `controller.ts` / `service.ts` / `dto/`
2. **フロントデータ取得**: `hooks/use-{entity}.ts` → `lib/api/{entity}.ts` → TanStack Query
3. **共有バリデーション**: `packages/shared/src/validators/` にZodスキーマ、フロント・バック両方で使用
4. **論理削除**: Prismaミドルウェアで `WHERE deleted_at IS NULL` を自動付与
5. **機能トグル**: 全モジュールのルートに機能チェックガード
6. **ページネーション**: チャットはカーソル方式、テーブルはオフセット方式
7. **ファイル管理**: 全ファイルは `files` テーブル経由 + R2アップロード
