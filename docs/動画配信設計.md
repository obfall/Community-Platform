# 動画配信設計 検討まとめ

## 前提条件

| 項目         | 内容                                                 |
| ------------ | ---------------------------------------------------- |
| 規模         | 小規模（月数十本アップロード、視聴数百回）           |
| アクセス制御 | 簡易的（ログインチェック + ランク制限程度。DRM不要） |
| 技術スタック | Next.js / NestJS / PostgreSQL / AWS / Cloudflare R2  |

---

## 配信方式の比較

### A. Cloudflare Stream（マネージド）

| 項目         | 内容                                                                 |
| ------------ | -------------------------------------------------------------------- |
| コスト       | $5/月基本料 + $5/1000分保存 + $1/1000分視聴（小規模で月$15〜20程度） |
| 実装工数     | 低（API連携のみ、数時間）                                            |
| 機能         | HLS自動変換・CDN内蔵・署名付きトークン・アダプティブ配信             |
| 運用負荷     | ほぼゼロ                                                             |
| 適合フェーズ | **MVP〜中規模**                                                      |

### B. R2 + 自前HLS（セルフホスト）

| 項目         | 内容                                                                      |
| ------------ | ------------------------------------------------------------------------- |
| コスト       | R2ストレージ $0.015/GB/月 + **帯域（エグレス）無料** + トランスコード実費 |
| 実装工数     | 高（FFmpegパイプライン構築、数日〜1週間）                                 |
| 構成         | アップロード → FFmpegでHLS変換 → R2にセグメント保存 → Cloudflare CDN配信  |
| 運用負荷     | FFmpegバージョン管理・エラー処理・キュー管理                              |
| 適合フェーズ | **スケール後（視聴数が月数万回を超えコストが問題になった時点）**          |

### C. 不採用とした方式

| 方式                               | 不採用理由                                                                     |
| ---------------------------------- | ------------------------------------------------------------------------------ |
| AWS MediaConvert + S3 + CloudFront | 小規模にはオーバースペック。トランスコードパイプラインの構築・管理コストが高い |
| Mux                                | 品質は高いが $0.007/分視聴で割高。小規模でもコスト > Cloudflare Stream         |
| YouTube/Vimeo埋め込み              | アクセス制御（ランク制限）が困難。UI制御不可。広告が入る                       |
| Bunny.net                          | 安いが、Cloudflare既採用なのに別CDNを追加する意味がない                        |

### トランスコード実行方式（自前HLSの場合）

| 方式                                 | コスト                     | 備考                                                     |
| ------------------------------------ | -------------------------- | -------------------------------------------------------- |
| NestJSワーカー + Bull Queue + FFmpeg | サーバー費のみ             | NestJSスタックに統合しやすい。小規模なら同一サーバーで可 |
| AWS ECS Fargate タスク               | 使った分だけ（$0.01〜/回） | 重い処理をオフロード。スケーラブル                       |
| AWS Lambda + FFmpeg Layer            | 使った分だけ               | 15分タイムアウト制限。長い動画は不向き                   |

## 推奨フェーズ戦略

```
Phase 1（MVP）: Cloudflare Stream
  → 実装最小、すぐリリース
  → 月$15〜20

Phase 2（スケール後）: 必要に応じて R2 + 自前HLS に移行
  → 視聴数が月数万回を超えてコストが問題になったら検討
  → テーブル設計が配信基盤に依存しないため、移行は容易
```

### Phase順序の根拠

「R2+自前HLS → Stream移行」ではなく「Stream → 自前HLS」を推奨する理由：

| 観点           | Stream先行                | 自前HLS先行                      |
| -------------- | ------------------------- | -------------------------------- |
| MVP開発速度    | **速い（API連携のみ）**   | 遅い（パイプライン構築必要）     |
| MVPコスト      | 月$15〜20で即稼働         | 開発工数 >> 月$15の節約          |
| 捨てるコード   | なし（APIコール差替のみ） | FFmpegパイプラインが丸ごと不要に |
| 移行タイミング | スケール時に必要なら着手  | 初期から重い実装                 |

## 開発環境の対応

Cloudflare Streamに専用の開発環境・サンドボックスは存在しない。

### Cloudflare Stream 料金体系

| 項目                   | 料金                                       |
| ---------------------- | ------------------------------------------ |
| 基本料金               | $5/月（1,000分のストレージ込み）           |
| 追加ストレージ         | $5/1,000分                                 |
| 配信                   | $1/1,000分視聴                             |
| Pro/Businessプラン付属 | 100分の無料ストレージ + 10,000分の無料配信 |

### 開発・テスト時の運用方針

| 環境             | 方式                                                                         |
| ---------------- | ---------------------------------------------------------------------------- |
| **ローカル開発** | Stream APIをモック化。ローカルのテスト動画ファイルで再生UIを開発             |
| **ステージング** | Streamアカウントを開発・ステージングで共有。短いテスト動画（10〜30秒）で検証 |
| **本番**         | Cloudflare Stream 本番接続                                                   |

月$5の最低料金内で小規模テストは十分収まる。

## テーブル設計

配信基盤に依存しない抽象化設計とし、Cloudflare Stream ↔ R2+自前HLS のどちらにも移行可能にする。

### `videos` テーブル 変更内容

既存の `video_url VARCHAR(500)` を削除し、以下のカラムに置き換え：

| カラム              | 型           | 制約                                                                               | 説明                          |
| ------------------- | ------------ | ---------------------------------------------------------------------------------- | ----------------------------- |
| `video_provider`    | VARCHAR(20)  | NOT NULL, CHECK IN ('cloudflare_stream','r2_hls')                                  | 配信基盤の識別子              |
| `video_external_id` | VARCHAR(200) | NOT NULL                                                                           | 外部ID（Stream ID or R2パス） |
| `playback_url`      | VARCHAR(500) | NULL                                                                               | HLS再生URL（キャッシュ）      |
| `stream_status`     | VARCHAR(20)  | NOT NULL, DEFAULT 'uploading', CHECK IN ('uploading','processing','ready','error') | 処理ステータス                |

既存の `thumbnail_url` はそのまま維持。

### 変更済みの `videos` テーブル全体

| カラム名           | 型           | 制約                                                                               | 説明                          |
| ------------------ | ------------ | ---------------------------------------------------------------------------------- | ----------------------------- |
| id                 | UUID         | PK, DEFAULT gen_random_uuid()                                                      | ID                            |
| category_id        | UUID         | FK → categories.id, NULL                                                           | カテゴリID                    |
| series_id          | UUID         | FK → video_series.id, NULL                                                         | シリーズID                    |
| title              | VARCHAR(200) | NOT NULL                                                                           | タイトル                      |
| description        | TEXT         | NULL                                                                               | 説明                          |
| video_provider     | VARCHAR(20)  | NOT NULL, CHECK IN ('cloudflare_stream','r2_hls')                                  | 配信基盤の識別子              |
| video_external_id  | VARCHAR(200) | NOT NULL                                                                           | 外部ID（Stream ID or R2パス） |
| playback_url       | VARCHAR(500) | NULL                                                                               | HLS再生URL（キャッシュ）      |
| stream_status      | VARCHAR(20)  | NOT NULL, DEFAULT 'uploading', CHECK IN ('uploading','processing','ready','error') | 処理ステータス                |
| thumbnail_url      | VARCHAR(500) | NULL                                                                               | サムネイルURL                 |

## 処理フロー

```
【アップロード】
管理者が動画アップロード
  → NestJS API
  → video_provider に応じて振り分け
     - cloudflare_stream: Stream API でアップロード（tusプロトコル）→ video_external_id 取得
     - r2_hls: R2 に原本保存 → FFmpeg で HLS 変換 → セグメントを R2 に保存
  → stream_status = 'uploading'

【トランスコード完了】
  → Webhook（Stream）or キュー完了通知（自前）
  → stream_status = 'ready' に更新
  → playback_url を保存

【再生】
ユーザーが動画再生リクエスト
  → NestJS API でログイン・ランク確認
  → video_provider に応じて再生URL生成
     - cloudflare_stream: 署名付きトークンを生成（有効期限付き）
     - r2_hls: 署名付き URL or 直接配信
  → フロントで HLS.js で再生
```
